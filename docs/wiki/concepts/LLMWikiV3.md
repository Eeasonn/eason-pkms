# LLM Wiki v3 知识编译系统

## 系统定位

**知识编译系统** —— 不是简单的文件存储，而是让 LLM 持续编译、维护、进化知识资产。

核心升级：从 Runtime RAG（提问时才检索）→ Compile-time Wiki（知识在编译时已被处理）。

## 三步编译法

### 第一步：浓缩
用剃刀法则：删掉这条信息会影响理解吗？不会就删。
输出：核心结论（不超过 3 条）+ 支撑每条结论的关键证据。

### 第二步：质疑
针对每条核心结论：
1. 依赖哪些前提假设？
2. 如果前提不成立，结论还成立吗？
3. 数据来源可靠吗？
4. 有没有反例或边界条件？

### 第三步：对标
1. 其他领域有没有类似现象？
2. 可以迁移到哪些场景？
3. 跨域关联 → 创建/更新概念条目

## 编译模式

| 模式 | 触发条件 | 动作 |
|------|---------|------|
| 加入 Wiki | "编译这篇文章" / "加入 Wiki" | 完整编译流程 |
| 加入待办 | "记个待办" / "待后续查阅" | 轻量记录到待办清单 |
| AI 主动识别 | 编译中发现需亲自尝试 | 主动询问是否记待办 |
| 问询触发 | 概念解释类问答 | 生成轻量概念页 |

## 标签体系

| 标签 | 含义 | 示例 |
|------|------|------|
| #工作-华为 | 华为工作场景相关 | 财经湖调度、数据治理口径 |
| #数据治理 | 专业领域 | 血缘追踪、元数据管理 |
| #AI应用 | AI+数据交叉 | ChatBI Agent、Semantic Weaver |
| #工具 | 具体工具 | DuckDB、dbt、DataHub |
| #vibecoding | 个人项目方向 | side project、创意编程 |
| #方法论 | 通用方法 | 三步编译法、PARA |

## 质量检查清单（强制）

- [ ] frontmatter 字段正确（date=编译日期，source/author/url正确）
- [ ] 概念文件核对（提取N个 = 生成N个）
- [ ] 索引统计正确（Total数字与实际一致）
- [ ] 日志一致性（描述与实际操作一致）
- [ ] 原文与资产完整性

## 健康检查（每周）

1. **一致性检查**：同一个概念在不同条目中的定义是否一致
2. **完整性检查**：每个概念条目是否包含所有必填字段
3. **孤岛检测**：找出入链和出链都少于 2 的页面
4. **跨目录一致性**：扫描风格规则有无遗漏或冲突
