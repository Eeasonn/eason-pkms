---

## 1. 核心理念

知识不是收藏，而是**编译**（compiled）。
- Raw Sources（原始材料）= 只读、不可变、保留完整元数据
- Wiki（知识层）= LLM 理解、提炼、关联、维护
- 你的角色 = 策展人（curator）：决定什么值得入库，什么要删除
- LLM 的角色 = 编译器（compiler）：理解、关联、维护、复习

## 3. 入库协议（Ingest Protocol）

### 3.1 触发条件
用户说以下任一表达即触发入库：
- "加入知识库"
- "这个值得记"
- "存一下"
- "入库"
- 链接 + 明确意图（如"这个文章帮我整理"）

### 3.2 必须保存的元数据（OKF Frontmatter）

每篇入库文档必须有 YAML frontmatter：

```yaml
---
type: Source | Entity | Concept | Project | Daily | Summary
title: 显示标题（原文标题或提炼）
description: 一句话总结（必须）
source_url: 原始链接（必须，保留原始）
source_title: 原始标题（必须，保留原始）
source_author: 作者（如有）
source_platform: 微信公众号 | 小红书 | 网页 | 书籍 | 播客 | 其他
ingest_date: 2026-07-11
access_date: 2026-07-11  ← 实际获取/阅读日期
tags: [标签1, 标签2]
status: inbox | processing | compiled | archived
review_count: 0
last_review: 2026-07-11
next_review: 2026-07-12  ← 复习日期
---
```

### 3.3 入库流程

**Step 1: 确认理解（用户验证）**
- LLM 读取原始内容
- LLM 提炼 3-5 个核心要点
- 询问用户："这是我的理解，对吗？你有补充吗？"
- 用户确认或修正

**Step 2: 分类决策**
- 判断类型：Source（来源摘要）/ Entity（实体）/ Concept（概念）/ Project（项目）
- 建议目录位置
- 询问用户确认

**Step 3: 写入 Raw（原始材料归档）**
- 写入 `raw/{platform}/YYYY-MM-DD-{slug}.md`
- 保留完整原文（或关键节选）
- 必须有完整元数据

**Step 4: 写入 Wiki（编译知识）**
- 写来源摘要：`wiki/sources/YYYY-MM-DD-{slug}.md`
- 更新相关实体页（`wiki/entities/`）
- 更新相关概念页（`wiki/concepts/`）
- 更新总目录 `wiki/index.md`
- 追加日志 `wiki/log.md`
- 创建复习任务 `review/due-YYYY-MM-DD.md`

**Step 5: 关联追问**
- "这和 `wiki/entities/XXX.md` 有关联，要链接吗？"
- "这个概念和你之前记的 `YYY` 有冲突/补充，要看一下吗？"

### 3.4 内容来源支持优先级

**当前支持（默认）：**
- ✅ 任何网页链接（通过 web_fetch）
- ✅ 微信公众号文章（通过 weixin-article-reader skill）
- ✅ 小红书笔记（通过 web_fetch + 解析）
- ✅ 飞书文档（通过 feishu-doc skill）
- ✅ 用户直接发送的文本/语音转文字
- ✅ **文本直通模式**（见 3.5 节）← 上班场景专用

**后续逐步扩充：**
- ⏳ Bilibili/YouTube 视频（通过 agent-reach yt-dlp）
- ⏳ 播客/音频（通过 agent-reach Groq Whisper）
- ⏳ PDF/论文（通过 nano-pdf skill）
- ⏳ 书籍章节（手动录入）
- ⏳ Twitter/X 线程（通过 agent-reach bird）

### 3.5 文本直通模式（Text Ingest Mode）

> **适用场景**：上班环境只能复制粘贴纯文本，不能反问、不能上传文件、不方便多轮对话。

#### 触发方式（三选一）

| 方式 | 说明 | 示例 |
|------|------|------|
| **标签触发** | 文本开头加 `#入库` 或 `#笔记` | `#笔记\n\n## DuckDB 优化技巧...` |
| **长度触发** | 文本 > 200 字且包含结构化内容（标题/列表/代码块） | 自动识别为笔记 |
| **显式触发** | 用户说"存一下""记笔记""入库" | 下一条消息执行 |

#### 处理规则（零反问，静默入库）

1. **不询问确认**：直接执行，不反问"要不要入库？"
2. **不询问分类**：自动推断类型（Source/Concept/Entity），默认存入 `inbox/`
3. **Markdown 解析**：自动解析文本中的 Markdown 结构（标题、列表、代码、引用等），提取结构化内容，存储时保留完整 Markdown 格式
4. **提取元数据**：
   - 标题：文本中第一个 `##` 或 `#` 标题，或前 30 字
   - 摘要：自动提取 1-2 句话核心要点
   - 标签：根据内容自动推断（#工作-华为 / #数据治理 / #AI / #工具 等）
   - 来源：标记为 `source_platform: 用户笔记`
5. **存储位置**：
   - 默认 → `inbox/YYYY-MM-DD-{slug}.md`
   - 如果用户明确说"直接入库" → 执行完整编译进 `wiki/`
6. **反馈方式**：
   - 只发一句确认："✅ 已丢进收件箱：`{标题}`（#{标签1} #{标签2}）"
   - 不展开、不追问、不打扰

**Markdown 格式兼容性说明**：
- 飞书消息不渲染 Markdown，但标记文本完整保留
- AI 后端能完整解析 `#`、`-`、`**`、`` ` ``、`>` 等标记
- 存到 PKMS 本地文件时格式完全恢复
- 表格/复杂代码块建议用 CSV 或分段贴

#### 示例流程

```
用户粘贴（飞书）：
#笔记

## DuckDB 窗口函数优化

今天发现 DuckDB 的窗口函数在数据量大时性能下降明显，
改用 CTE 分段处理后速度提升 10x。

关键步骤：
1. 先按分区列排序
2. 使用 ROW_NUMBER() 代替自连接
3. 最后 UNION ALL 合并

注意：内存要设置足够大，否则还是慢。

---

AI（自动识别 → 零反问 → 静默入库）：
✅ 已丢进收件箱：DuckDB 窗口函数优化（#工具 #数据工程）
```

#### 事后处理

- 用户下班后可说"处理今天的收件箱" → 执行完整 Ingest Protocol（确认理解 → 分类 → 编译 → 复习）
- 或用户随时说"编译刚才那条笔记" → 单独处理

## 4. 复习协议（Review Protocol）

### 4.1 间隔重复（Spaced Repetition）

每次复习后更新 `next_review`：

| 复习次数 | 下次间隔 | 复习方式 |
|---------|---------|---------|
| 0（新入库） | 1 天后 | LLM 提问核心要点 |
| 1 | 3 天后 | LLM 提问关联知识 |
| 2 | 7 天后 | 用户主动回忆 + LLM 补充 |
| 3 | 14 天后 | 快速回顾 |
| 4+ | 30 天后 | 定期维护 |

### 4.2 复习流程

1. LLM 从 `review/due-*.md` 读取当天到期的知识
2. 向用户提问（飞书推送）
3. 用户回复或忽略
4. 更新 `review_count` 和 `next_review`
5. 如果用户表示"已遗忘"，重置间隔为 1 天

## 5. Daily Log 协议

### 5.1 触发

Cron：工作日 **17:00**

LLM 主动提问：
```
📋 工作日报时间

1. 今天主要做了什么？（项目/任务）
2. 遇到什么困难/卡点？
3. 有什么新想法或启发？
4. 明天计划？

（语音/打字都行，简单几句即可）
```

**回复渠道**：
- 正常情况下：M5（飞书）或手机（飞书）
- 公司电脑不方便时：Kimi 纯文本回复（我也会接收）

### 5.2 处理流程

1. 用户回复后，LLM 整理为结构化 markdown
2. 写入 `wiki/daily/YYYY-WXX.md`（按周汇总）
3. 追加到 `wiki/log.md`
4. 提取模式：
   - 重复困难 → 建议主动解决
   - 和目标关联 → 更新目标跟踪
   - 新想法 → 建议入库或创建 TODO

## 9. 思考三问法（Ingest Core）

> 入库任何知识时，LLM 必须先回答这三个问题，再写入 wiki。
> 这是内化（compiled）的核心，不是简单的收藏。

### 9.1 三问模板

入库时，LLM 先在内部完成这三问，再写文件：

**Q1: 哪句话打动了我？（What struck me?）**
- 找到文章/材料中最触动你的那句话/段落
- 不是总结，是原汁原味的"击中感"

**Q2: 为什么它打动我？（Why does it resonate?）**
- 这句话关联了我什么经验/困惑/目标？
- 它挑战/印证/补充了我什么认知？

**Q3: 接下来做什么？（What to do next?）**
- 立刻能做的一件小事（5分钟内）
- 或者一个开放性问题（持续思考）

### 9.2 输出位置

三问答案写入对应 `wiki/sources/YYYY-MM-DD-{slug}.md` 的开头：

```markdown
## 💡 内化三问

**Q1 打动我的那句话：**
> "[原文引用]"

**Q2 为什么它打动我：**
[2-3 句话，关联自身]

**Q3 接下来做什么：**
- [可执行的下一小步]
- [持续思考的问题]
```

### 9.3 注意

- 三问是 **AI 内部思考**，不是问用户
- 用户说"加入知识库"时，AI 读取原文后先完成三问，再执行入库流程
- 三问帮助知识从"收藏"走向"编译"

## 10. 学习→项目转化机制

### 10.1 双轨识别

**轨道 A：AI 自动识别（主动推送）**
- AI 在复习/入库时发现某知识已成熟
- 主动向用户推送：`这条知识可以转化为项目了，要试试吗？`

**轨道 B：用户手动标记**
- 用户在任意知识旁加 `#可行动` 标签
- AI 检测到后自动创建项目卡片

### 10.2 项目创建流程

1. **触发**：AI 推送 或 用户标记 `#可行动`
2. **讨论**：AI 提出项目草案（名称、目标、第一步）
3. **确认**：用户确认或修改
4. **写入**：`wiki/projects/{项目名}.md`
5. **关联**：在原知识页添加链接到项目页

### 10.3 项目页结构

```markdown
---
type: Project
title: [项目名称]
description: [一句话目标]
status: ideation | active | paused | completed
start_date: 2026-07-12
tags: [#AI #数据工程 ...]
related_knowledge: [[source slug]], [[concept slug]]
---

## 项目目标
[具体、可衡量的目标]

## 第一步行动
[今天就能做的第一件事]

## 进度记录
- 2026-07-12: 项目创建
```
